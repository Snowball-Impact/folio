# FOLIO 와이어프레임

이 문서는 현재 코드베이스 기준의 화면 구조를 설명한다.

## 디자인 방향

- 데이터 분석 프로젝트를 포트폴리오 자산으로 축적하고 탐색하는 MVP
- PC 화면 우선
- 모바일 대응은 하되, PC 레이아웃을 과하게 좁히지 않는다.
- 카드와 링크는 기본 브라우저 링크 스타일을 제거하고, 서비스 UI처럼 보이게 한다.
- 카테고리는 사용하지 않는다. 탐색과 분류는 태그 중심이다.

## 전역 헤더

```text
┌──────────────────────────────────────────────┐
│ FOLIO                              [☰]       │
│ 발표로 끝나지 않는 프로젝트                    │
└──────────────────────────────────────────────┘
```

- 메뉴는 햄버거 버튼 안의 작은 드롭다운으로 표시한다.
- 비로그인 메뉴: 홈, 둘러보기, 로그인, 소개
- 로그인 메뉴: 홈, 둘러보기, 프로젝트 제출, 내 포트폴리오, 프로필, 소개, 로그아웃
- 회원가입은 상단 메뉴에 노출하지 않는다.
- 회원가입 페이지는 로그인 화면의 링크와 `?page=Sign Up` 라우트로 접근한다.

## 1. Home

### 목적

서비스 핵심 메시지와 인기 프로젝트 탐색 진입을 제공한다.

### 현재 구조

```text
┌──────────────────────────────────────────────┐
│ AI 시대에도                                  │
│ 휴먼 인사이트는 자산이다.                     │
│                                              │
│ 데이터 분석 프로젝트를 포트폴리오로             │
│ 축적하고 공유하세요.                          │
│                                              │
│                         [프로젝트 인사이트]    │
└──────────────────────────────────────────────┘

검색
[프로젝트명, 태그, 인사이트로 검색] [🔎]

#인기태그 #인기태그 #인기태그

인기 프로젝트
[카드] [카드] [카드] [카드]
```

### 규칙

- CTA 버튼은 별도로 두지 않는다.
- 홈 카드 전체가 상세 페이지 링크다.
- 검색어는 Gallery의 `q` 쿼리로 이어진다.
- 태그 목록은 실제 등록 프로젝트의 인기 태그에서 가져온다.

## 2. Gallery

### 목적

공개 프로젝트를 검색, 태그 필터, 정렬로 탐색한다.

### 현재 구조

```text
┌──────────────────────────────────────────────┐
│ 프로젝트 갤러리                                │
│ 데이터 분석 포트폴리오와 인사이트를 탐색하세요. │
└──────────────────────────────────────────────┘

[검색어 입력]

태그 필터: [전체] [태그] [태그] ...
정렬: [최신순 | 조회수순 | 좋아요순]

총 N개의 프로젝트

┌──────────────────────────────────────────────┐
│ 썸네일 │ 프로젝트명                           │
│        │ 한 줄 소개                           │
│        │ 작성자                               │
│        │                         #태그 #태그  │
│        │                         조회 · 좋아요│
└──────────────────────────────────────────────┘
```

### 규칙

- 카드 전체가 상세 페이지 링크다.
- 조회수, 좋아요, 태그는 카드 우하단에 둔다.
- 카테고리 필터는 없다.
- 좋아요 수는 `likes` 테이블을 기준으로 계산한다.

## 3. Project Detail

### 목적

프로젝트의 문제 정의, 사용 데이터, 분석 과정, 핵심 인사이트와 첨부 자료를 보여준다.

### 현재 구조

```text
┌──────────────────────────────────────────────┐
│ 프로젝트 제목                                  │
│ 작성자 · 등록일                                │
│                                      #태그     │
│                                      조회/좋아요│
└──────────────────────────────────────────────┘

왼쪽 본문                         오른쪽 사이드
AI 요약                           대시보드
문제 정의                         첨부 파일 및 링크
사용 데이터
분석 과정
핵심 인사이트

[목록으로 돌아가기]
```

### 규칙

- 상세 페이지는 Gallery 메뉴 안에서 `project_id` 쿼리로 렌더링한다.
- 대시보드 제목은 `대시보드`다.
- Power BI 문구는 상세 화면 제목에 노출하지 않는다.
- 대시보드 iframe은 우측 사이드에서 16:12 비율을 사용한다.

## 4. Submit / Edit Project

### 목적

로그인한 사용자가 프로젝트를 등록하거나 수정한다.

### 현재 구조

```text
프로젝트 정보

왼쪽 주요 입력
- 프로젝트명
- 한 줄 소개
- 태그
- 자유 입력 본문 편집기

오른쪽 선택 입력
- Power BI Embed URL
- 보고서 URL
- GitHub URL
- 썸네일 URL
- AI 요약
```

### 규칙

- 프로젝트 본문은 자유 입력이다.
- 섹션 제목을 유지하면 상세 화면에서 `문제 정의`, `사용 데이터`, `분석 과정`, `핵심 인사이트`로 나뉜다.
- 필수 입력값은 프로젝트명과 본문이다.
- 태그는 쉼표 기준으로 최대 10개 저장한다.
- 카테고리 입력은 없다.
- 링크 및 요약은 접힌 고급 영역에 둔다.

## 5. My Portfolio

### 목적

로그인 사용자가 자신이 등록한 프로젝트를 관리한다.

### 기능

- 내 프로젝트 목록 표시
- 새 프로젝트 등록 이동
- 상세 보기
- 수정
- 삭제 확인 체크 후 삭제

## 6. Profile

### 목적

사용자의 기본 프로필과 등록 프로젝트 통계를 보여준다.

### 현재 표시 정보

- 이름
- 이메일
- 기관
- 소개
- 등록 프로젝트 수
- 총 조회수
- 작성 프로젝트 목록

## 7. Login / Sign Up

### 목적

Supabase Auth 기반 로그인과 회원가입을 제공한다.

### 현재 구조

```text
┌──────────────────────────────────────────────┐
│ 로그인 또는 회원가입                           │
│ 짧은 설명                                     │
├──────────────────────────────────────────────┤
│ 이메일                                        │
│ 비밀번호                                      │
│ [로그인 또는 회원가입]                         │
└──────────────────────────────────────────────┘
```

### 규칙

- 로그인/회원가입은 중앙의 작은 인증 모듈로 배치한다.
- 인증 페이지에 별도 대형 히어로는 없다.
- 회원가입 페이지 하단에는 로그인 전환 링크를 두지 않는다.
- 회원가입 페이지의 인증 메일 재발송은 접힌 보조 영역으로 둔다.
- 정상 입력마다 성공 메시지를 띄우지 않고, 문제 상황만 오류/경고로 보여준다.
- 비밀번호 입력폼은 다른 입력폼과 같은 디자인을 유지한다.

### 현재 구조

- 왜 만들었나요?
- 핵심 철학

## Interaction Rules

### View Count

- 상세 페이지 진입 시 조회수가 증가한다.
- 같은 세션에서는 같은 프로젝트 조회수를 중복 증가시키지 않는다.

### Likes

- 좋아요 데이터는 `likes` 테이블에 저장된다.
- 좋아요 수는 프로젝트 목록과 상세 메타에 표시된다.
- 좋아요순 정렬은 실제 likes 데이터를 기반으로 계산한다.

### Public Read

- 공개 프로젝트는 비로그인 사용자도 읽을 수 있다.
- Supabase JWT 만료로 public read가 실패하면 anon key로 복구한다.

## 핵심 UX 원칙

FOLIO는 프로젝트를 단순히 저장하는 공간이 아니다.

사용자가 다음 질문에 빠르게 답할 수 있어야 한다.

1. 이 프로젝트는 어떤 문제를 다루는가?
2. 어떤 데이터를 사용했는가?
3. 어떤 분석 과정을 거쳤는가?
4. 어떤 인사이트를 얻었는가?
5. 작성자는 어떤 역량을 보여주는가?
