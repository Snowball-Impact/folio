# FOLIO 와이어프레임

이 문서는 현재 코드베이스 기준의 화면 구조를 설명한다.

## 디자인 방향

- 데이터 분석 프로젝트를 포트폴리오 자산으로 축적하고 탐색하는 MVP
- PC 화면 우선
- 모바일 대응은 하되, PC 레이아웃을 과하게 좁히지 않는다.
- 카드와 링크는 기본 브라우저 링크 스타일을 제거하고, 서비스 UI처럼 보이게 한다.
- 카테고리는 사용하지 않는다. 탐색과 분류는 태그 중심이다.

## 전역 헤더

```text
┌──────────────────────────────────────────────┐
│ FOLIO                              [☰]       │
│ 발표로 끝나지 않는 프로젝트                    │
└──────────────────────────────────────────────┘
```

- 메뉴는 햄버거 버튼 안의 작은 드롭다운으로 표시한다.
- 비로그인 메뉴: 홈, 로그인
- 로그인 메뉴: 홈, 프로젝트 제출, 마이 페이지, 로그아웃
- 회원가입은 상단 메뉴에 노출하지 않는다.
- 회원가입 페이지는 로그인 화면의 링크와 `?page=Sign Up` 라우트로 접근한다.

## 1. Home

### 목적

서비스 핵심 메시지와 프로젝트 탐색 경험을 함께 제공한다.

### 현재 구조

```text
┌──────────────────────────────────────────────┐
│ AI 시대에도                                  │
│ 휴먼 인사이트는 자산이다.                     │
│                                              │
│ 데이터 분석 프로젝트를 포트폴리오로             │
│ 축적하고 공유하세요.                          │
│                                              │
│                         [프로젝트 인사이트]    │
└──────────────────────────────────────────────┘

프로젝트 탐색
[프로젝트명, 태그, 인사이트로 검색] [검색 적용]

태그 필터: [전체] [태그] [태그] ...
정렬: [최신순 | 조회수순 | 좋아요순]

프로젝트 총 N개
[카드] [카드] [카드] [카드]
```

### 규칙

- CTA 버튼은 별도로 두지 않는다.
- 홈 카드 전체가 상세 페이지 링크다.
- 검색어, 태그, 정렬은 `검색 적용` 시 Home의 탐색 결과를 갱신한다.
- 입력 도중 전체 앱과 Supabase 조회가 반복되지 않도록 필터를 form으로 묶는다.
- 검색과 태그 필터는 공개 프로젝트 전체 페이지를 대상으로 적용한다.
- 태그 목록은 실제 등록 프로젝트의 인기 태그에서 가져온다.
- 프로젝트가 1개일 때는 가로형 featured 카드로 표시한다.

## 2. Gallery

### 목적

기존 링크 호환성을 위한 레거시 라우트다. 현재 공개 프로젝트 탐색은 Home에서 처리한다.

### 현재 구조

```text
?page=Gallery
↓
?page=Home 으로 이동
```

### 규칙

- `q`, `tag`, `project_id` 같은 기존 쿼리 값은 보존해서 Home으로 넘긴다.
- 새 UI 작업은 Home 탐색 패널 기준으로 진행한다.

## 3. Project Detail

### 목적

프로젝트의 문제 정의, 사용 데이터, 분석 과정, 핵심 인사이트와 첨부 자료를 보여준다.

### 현재 구조

```text
┌──────────────────────────────────────────────┐
│ 프로젝트 제목                                  │
│ 작성자 · 등록일                                │
│                                      #태그     │
│                                      조회/좋아요│
└──────────────────────────────────────────────┘

왼쪽 본문                         오른쪽 사이드
문제 정의                         대시보드
사용 데이터                       첨부 파일 및 링크
분석 과정
핵심 인사이트

[목록으로 돌아가기]
```

### 규칙

- 상세 페이지는 Home 안에서 `project_id` 쿼리로 렌더링한다.
- 대시보드 제목은 `대시보드`다.
- Power BI 문구는 상세 화면 제목에 노출하지 않는다.
- 대시보드 iframe은 우측 사이드에서 16:12 비율을 사용한다.
- 좋아요 버튼은 상세 메타 영역에 표시하며, 비로그인 사용자는 로그인 화면으로 이동한다.
- 대시보드나 첨부 링크가 없으면 본문을 전체 폭으로 표시한다.
- 모바일에서는 본문 다음에 대시보드와 첨부 링크를 세로로 배치한다.
- 목록 복귀와 좋아요는 세션을 유지하는 Streamlit 버튼을 사용한다.

## 4. Submit / Edit Project

### 목적

로그인한 사용자가 프로젝트를 등록하거나 수정한다.

### 현재 구조

```text
1. 기본 정보 + 실시간 카드 미리보기
- PC: 하나의 카드 안에서 왼쪽에 프로젝트명·한 줄 소개·태그 입력, 오른쪽에 Home 카드 미리보기
- 모바일: 기본 정보 아래에 카드 미리보기를 1열로 배치
- 카드 커버는 Home과 같은 16:9 비율

2. 프로젝트 내용
- 자유 입력 본문 편집기

3. 관련 결과물 링크
- BI Platform Embed URL
- GitHub URL
- ETC URL
- PC에서는 1행 3열, 모바일에서는 1열로 배치
```

### 규칙

- 프로젝트 본문은 자유 입력이다.
- 섹션 제목을 유지하면 상세 화면에서 `문제 정의`, `사용 데이터`, `분석 과정`, `핵심 인사이트`로 나뉜다.
- 필수 입력값은 프로젝트명과 본문이다.
- 태그는 쉼표 기준으로 최대 10개 저장한다.
- 카테고리 입력은 없다.
- 기본 정보·미리보기, 프로젝트 내용, 관련 결과물 링크를 단계별 섹션으로 구분한다.
- 선택 URL은 입력 위치에서 형식을 바로 안내한다.
- 카드 미리보기는 기본 정보 입력과 함께 항상 표시하고 rerun 시 실시간 갱신한다.
- 저장 전 새로고침 시 작성 내용이 사라질 수 있음을 안내한다.

## 5. My Page

### 목적

로그인 사용자가 자신의 프로필과 등록 프로젝트를 한 화면에서 관리한다. `My Portfolio`, `Profile`은 과거 별도 화면이었으나 지금은 `My Page`로 통합됐고, 두 라우트는 호환을 위해 `My Page`로 리다이렉트만 한다.

### 기능

- 프로필 카드: 이름, 이메일, 기관, 소개, 등록 프로젝트 수·공개 수·누적 조회·총 좋아요 통계
- 프로필 편집
- 내 프로젝트 목록 표시, 새 프로젝트 등록 이동
- 프로젝트별 상세 보기 / 수정 / 삭제(확인 대화상자에서 최종 삭제)
- 프로젝트 정보와 보기·수정·삭제 액션을 하나의 관리 카드로 묶음

## 6. Login / Sign Up

### 목적

Supabase Auth 기반 로그인과 회원가입을 제공한다.

### 현재 구조

```text
┌──────────────────────────────────────────────┐
│ 로그인 또는 회원가입                           │
│ 짧은 설명                                     │
├──────────────────────────────────────────────┤
│ 이메일                                        │
│ 비밀번호                                      │
│ [로그인 또는 회원가입]                         │
└──────────────────────────────────────────────┘
```

### 규칙

- 로그인/회원가입은 중앙의 작은 인증 모듈로 배치한다.
- 인증 페이지에 별도 대형 히어로는 없다.
- 회원가입 페이지 하단에는 로그인으로 돌아가는 전환 링크를 둔다(로그인 화면의 "회원가입하기"와 대칭).
- 회원가입 페이지의 인증 메일 재발송은 접힌 보조 영역으로 둔다.
- 정상 입력마다 성공 메시지를 띄우지 않고, 문제 상황만 오류/경고로 보여준다.
- 비밀번호 입력폼은 다른 입력폼과 같은 디자인을 유지한다.


## Interaction Rules

### View Count

- 상세 페이지 진입 시 조회수가 증가한다.
- 같은 세션에서는 같은 프로젝트 조회수를 중복 증가시키지 않는다.

### Likes

- 좋아요 데이터는 `likes` 테이블에 저장된다.
- 좋아요 수는 프로젝트 목록과 상세 메타에 표시된다.
- 좋아요순 정렬은 실제 likes 데이터를 기반으로 계산한다.

### Public Read

- 공개 프로젝트는 비로그인 사용자도 읽을 수 있다.
- Supabase JWT 만료로 public read가 실패하면 anon key로 복구한다.

## 핵심 UX 원칙

FOLIO는 프로젝트를 단순히 저장하는 공간이 아니다.

사용자가 다음 질문에 빠르게 답할 수 있어야 한다.

1. 이 프로젝트는 어떤 문제를 다루는가?
2. 어떤 데이터를 사용했는가?
3. 어떤 분석 과정을 거쳤는가?
4. 어떤 인사이트를 얻었는가?
5. 작성자는 어떤 역량을 보여주는가?
