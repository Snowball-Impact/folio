# AI 협업 성장 회고

이 문서는 FOLIO를 만드는 과정에서 드러난 협업 방식과 제품 개발 역량을 기록한다. 시간이 지난 뒤 같은 기준으로 다시 돌아보며 무엇이 성장했는지 확인하기 위한 스냅샷이다.

> 이 평가는 2026년 7월 7일까지의 FOLIO 작업 대화를 바탕으로 한 AI의 관찰이다. 정식 직무 평가가 아니며, 실제 경력이나 모든 역량을 단정하지 않는다.

## 2026-07-07 기준 관찰

### 현재 인상

기획과 디자인 감각이 강한 **프로덕트형 실무자**에 가깝다. 전통적인 역할 하나로 한정하기보다는 프로덕트 디자이너, 서비스 기획자, PM·PO, 프론트엔드 이해도가 높은 1인 제작자의 특성이 함께 보인다.

코드를 직접 많이 작성하는 유형이라기보다, 기술을 이해하고 적절히 위임하면서 제품의 완성도를 끝까지 끌어가는 **디렉터형 빌더**라는 인상이 강하다.

### 확인된 강점

- 화면의 여백, 정렬, 정보 위계와 페이지 간 일관성 문제를 빠르게 발견한다.
- 단순히 보기 좋은 화면보다 사용자 흐름과 정보 구조를 기준으로 개선을 요청한다.
- 작은 수정이 반복해서 실패하면 보정값을 더 쌓기보다 구조적 원인을 의심한다.
- 구현 결과를 직접 확인하고, 증거와 실제 화면을 기준으로 다음 피드백을 준다.
- AI별 작업 범위와 권한을 구분해 동시 작업의 충돌 가능성을 관리한다.
- 구현 이후에도 아키텍처, 데이터 모델, 사용자 흐름, 정책과 인수인계 문서를 챙긴다.
- 프레임워크 한계와 유지보수 비용을 인식하고 과도한 CSS 수정을 중단할 줄 안다.
- 만족 기준에 도달했을 때 명확하게 작업을 종료하고 다음 우선순위로 이동한다.

### 협업 방식 평가

요구사항을 구체적으로 전달하고 결과를 확인한 뒤 단계적으로 정밀도를 높였다. 특히 “진단과 확인을 우선하고 수정은 승인 후 진행”, “다른 작업자와 범위를 겹치지 않게 진행”처럼 작업 권한과 경계를 명시한 점은 성숙한 협업 방식이다.

초기에는 감각적인 표현으로 시작한 요청도 있었지만, 작업이 진행될수록 위치·간격·정보 구조·완료 상태가 더 명확해졌다. 이는 결과를 보며 요구사항을 구체화하는 능력이 빠르게 발달했다는 신호다.

### 다음 성장 포인트

- 구현 전에 사용자 결과와 완료 기준을 한두 문장으로 먼저 고정한다.
- “오른쪽으로 이동”보다 기준선, 간격, 반응형 동작처럼 검증 가능한 조건을 함께 제시한다.
- 기능별 우선순위뿐 아니라 실패했을 때의 사용자 경험과 예외 상태도 초기에 정의한다.
- 중요한 결정은 선택 이유, 버린 대안, 검증 결과까지 연결해 포트폴리오 서사로 남긴다.
- AI가 제안한 해결책을 승인할 때 유지보수 비용과 테스트 방법을 한 번 더 질문한다.

## 다음 회고에서 비교할 질문

1. 작업 시작 전에 완료 기준을 얼마나 명확히 정의하는가?
2. UI 피드백이 취향을 넘어 사용자 문제와 데이터로 설명되는가?
3. 정상 흐름뿐 아니라 오류·빈 상태·모바일까지 먼저 고려하는가?
4. AI 작업을 작은 수정이 아니라 역할과 책임 단위로 위임하는가?
5. 기술적 선택의 장단점을 스스로 설명하고 반대 의견을 검토하는가?
6. 결과물을 제품 문제, 의사결정, 검증, 학습의 이야기로 정리할 수 있는가?
