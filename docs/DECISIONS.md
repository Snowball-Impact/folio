# FOLIO 설계 결정 기록

이 문서는 FOLIO를 만들며 선택한 주요 기술·제품 결정을 ADR(Architecture Decision Record) 형식으로 요약한다. 결정은 현재 코드의 맥락을 설명하며, 조건이 바뀌면 다시 검토할 수 있다.

## ADR-001. MVP 프레임워크로 Streamlit을 사용한다

- **상태**: 채택
- **맥락**: 데이터 분석가가 프로젝트와 대시보드를 빠르게 등록하고 보여주는 MVP가 필요했다.
- **결정**: Python 생태계와 짧은 UI 개발 주기를 활용할 수 있는 Streamlit을 사용한다.
- **결과**:
  - 데이터·분석 도구와 같은 언어로 빠르게 개발한다.
  - 서버 렌더링과 rerun 모델을 고려해 상태를 설계해야 한다.
  - 복잡한 인터랙션과 세밀한 DOM 제어에는 한계가 있으므로 key 기반 CSS와 실측 검증이 필요하다.
- **재검토 조건**: 협업 편집, 고도화된 클라이언트 상태, SEO 또는 대규모 트래픽이 핵심 요구가 될 때.

## ADR-002. URL query 기반 단일 라우터를 사용한다

- **상태**: 채택
- **맥락**: 공개 프로젝트 상세 URL을 공유하면서 인증 세션을 안정적으로 보존해야 했다.
- **결정**: 파일 기반 멀티페이지 대신 `?page=`와 `?project_id=`를 `folio_app/app.py`에서 해석한다.
- **결과**:
  - 공통 헤더·인증 복구·온보딩 검사를 한 진입점에서 처리한다.
  - 내부 상태 변경 이동은 `navigate()`로 통일한다.
  - 브라우저 전체 reload를 일으키는 HTML 링크는 공개 카드 탐색 외에는 사용하지 않는다.

## ADR-003. 인증 토큰은 세션 상태와 암호화 쿠키에 함께 둔다

- **상태**: 채택
- **맥락**: Streamlit rerun 중 로그인 상태를 유지하고 브라우저 재접속 때 세션을 복구해야 했다.
- **결정**: 현재 사용자와 토큰은 `session_state`, 재접속용 토큰은 `EncryptedCookieManager`에 저장한다.
- **결과**:
  - 앱 시작 시 쿠키 토큰으로 Supabase 세션을 복구한다.
  - 복구 실패 시 토큰을 제거하고 공개 페이지는 계속 보여준다.
  - 보호 페이지는 Login으로 이동해 재인증을 안내한다.
  - 로그아웃은 서버 세션, 쿠키, 세션별 Supabase client를 모두 정리한다.

## ADR-004. Supabase client를 사용자 세션별로 격리한다

- **상태**: 채택
- **맥락**: Auth 상태를 가진 client를 전역 `st.cache_resource`로 공유하면 사용자 간 JWT가 섞일 위험이 있다.
- **결정**: Supabase client를 `session_state`에 보관해 Streamlit 세션 안에서만 재사용한다.
- **결과**:
  - 사용자별 인증 상태가 격리된다.
  - 로그아웃과 만료 JWT 처리 시 client를 폐기한다.
  - 작성자 mutation 직전 세션을 재연결하고 PostgREST에 최신 JWT를 명시 적용한다.

## ADR-005. 데이터 권한의 최종 경계는 RLS로 둔다

- **상태**: 채택
- **맥락**: UI에서 버튼을 숨기는 것만으로는 직접 API 호출을 막을 수 없다.
- **결정**: 공개 읽기, 작성자 CRUD, 좋아요, 정책 동의를 PostgreSQL RLS로 강제한다.
- **결과**:
  - 앱 검증 실패가 있어도 DB가 최종 권한을 확인한다.
  - 공개 프로필은 최소 컬럼만 제공하는 view를 사용한다.
  - 로컬 코드 변경과 원격 RLS 적용을 별도 완료 조건으로 관리한다.

## ADR-006. 프로젝트 본문은 제한된 HTML로 저장한다

- **상태**: 채택
- **맥락**: Quill의 서식을 유지하면서 XSS와 위험한 링크를 차단해야 했다.
- **결정**: 허용 태그·속성·프로토콜 기반 sanitizer를 저장과 출력 양쪽에 적용한다.
- **결과**:
  - 제목, 문단, 목록, 강조, 안전한 링크 등 작성 서식을 유지한다.
  - script, event handler, 위험 프로토콜을 제거한다.
  - 미리보기와 상세 출력이 같은 정제 규칙을 사용한다.

## ADR-007. 태그를 핵심 탐색 축으로 사용한다

- **상태**: 채택
- **맥락**: 초기 MVP에서 고정 카테고리는 프로젝트의 다양한 분석 주제를 충분히 표현하지 못했다.
- **결정**: 카테고리를 제거하고 최대 10개의 태그, 검색, 정렬을 결합한다.
- **결과**:
  - 데이터 소스·도구·도메인을 유연하게 표현한다.
  - 인기 태그는 공개 프로젝트 캐시에서 계산한다.
  - 검색은 제목, 소개, 본문 섹션과 태그를 함께 대상으로 한다.

## ADR-008. 공개 목록 원본을 짧게 캐시하고 mutation 후 무효화한다

- **상태**: 채택
- **맥락**: Streamlit rerun마다 프로젝트·프로필·좋아요를 다시 요청하면 지연과 API 비용이 커진다.
- **결정**: 프로젝트 30초, 공개 프로필 60초, 좋아요 수 15초 캐시를 사용한다.
- **결과**:
  - 검색·태그 필터는 캐시된 원본의 복사본에 적용한다.
  - CRUD, 조회수, 좋아요 변경 시 관련 캐시를 즉시 비운다.
  - 캐시 지연과 데이터 최신성의 균형을 명시적으로 관리한다.

## ADR-009. 프로필과 포트폴리오 관리를 My Page로 통합한다

- **상태**: 채택
- **맥락**: Profile과 My Portfolio가 동일한 사용자·프로젝트 데이터를 반복 조회하고 탐색 경로를 분산시켰다.
- **결정**: 프로필, 통계, 프로젝트 보기·수정·삭제를 My Page 하나에 배치한다.
- **결과**:
  - 사용자 데이터 조회와 관리 동선이 단순해졌다.
  - 이전 URL은 호환 리다이렉트로 유지한다.
  - 모바일에서는 관리 액션을 프로젝트 카드 하단 가로 영역으로 전환한다.

## ADR-010. CSS를 화면 영역별 모듈로 관리한다

- **상태**: 채택
- **맥락**: 단일 `styles.py`가 2,400줄 이상으로 커지며 중복·죽은 선택자와 캐스케이드 충돌이 발생했다.
- **결정**: `styles/` 패키지에서 화면 영역별 CSS 상수를 관리하고 고정 순서로 한 번 주입한다.
- **결과**:
  - 스타일 소유권과 탐색성이 명확해졌다.
  - `_SECTIONS` 순서는 동일 specificity 충돌의 타이브레이커이므로 변경 시 검토한다.
  - 새 선택자는 실제 렌더링 클래스·key와 대조한 뒤 추가한다.

## ADR-011. 좋아요와 조회수는 서로 다른 모델을 사용한다

- **상태**: 채택
- **맥락**: 좋아요는 사용자별 중복 방지가 필요하고 조회수는 빠른 원자 증가가 필요하다.
- **결정**:
  - 좋아요는 `(project_id, user_id)` 관계 테이블로 저장한다.
  - 조회수는 `projects.view_count`에 두고 RPC로 증가시킨다.
- **결과**:
  - 좋아요 여부와 취소를 정확히 추적한다.
  - 조회수 증가 경쟁 조건을 DB 함수로 줄인다.
  - 동일 브라우저 세션의 중복 조회는 애플리케이션 상태로 억제한다.
