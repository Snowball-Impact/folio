# 실제 Supabase 통합 검증 기록

이 문서는 단위 테스트만으로 확인할 수 없는 FOLIO와 원격 Supabase 간 실제 동작을 기록한다. 테스트 계정의 이메일·비밀번호와 같은 민감 정보는 남기지 않는다.

## 2026-07-07 검증

### 방법과 범위

- 기존 온보딩 완료 테스트 계정으로 실제 서비스에 로그인했다.
- Selenium으로 브라우저를 직접 조작해 사용자 흐름과 DB 반영 결과를 함께 확인했다.
- 검증 중 생성한 테스트 프로젝트는 모두 삭제했다.
- 실제 이메일 발송과 새 계정 생성이 필요한 회원가입 흐름은 사용자 승인 없이 진행하지 않았다.

### 결과

| 항목 | 결과 | 확인 내용 |
|---|---|---|
| 로그인 | 통과 | 실제 Supabase 계정으로 정상 로그인 |
| 프로젝트 등록 | 통과 | Quill 본문을 포함한 프로젝트 생성 |
| 홈 검색·태그 | 통과 | 생성한 공개 프로젝트 검색 및 태그 노출 |
| 좋아요 추가·취소 | 통과 | 화면 동작 및 DB 카운트의 정확한 반영 |
| 공개→비공개 전환 | 통과 | 저장 후 실제 DB 상태 변경 |
| 비공개 탐색 제한 | 통과 | 전환 후 홈 검색 결과에서 제외 |
| 작성자 비공개 열람 | 통과 | 작성자는 비공개 프로젝트 열람 가능, RLS 정상 |
| 같은 Streamlit 세션의 조회수 중복 방지 | 통과 | 같은 세션에서 반복 조회해도 한 번만 증가 |

핵심 CRUD, 공개 범위 RLS, 좋아요 흐름은 실제 원격 환경에서 정상 동작했다.

### 발견 사항: 조회수의 측정 단위

현재 조회수 중복 방지는 `session_state`의 `viewed_<project_id>` 값을 사용한다. 같은 Streamlit 세션에서는 의도대로 동작하지만 브라우저 하드 리로드 후에는 새 세션이 열려 조회수가 다시 증가했다. 실제 확인에서는 같은 프로젝트를 반복 리로드했을 때 조회수가 `2 → 3 → 4`로 증가했다.

공개 프로젝트 카드 전체 클릭에는 탐색 UX를 위해 HTML `<a>` 링크를 허용한다. 이 이동은 브라우저 전체 리로드를 일으킬 수 있으므로 일반적인 카드 진입 경로에서도 기존 세션 기반 중복 방지의 실효성이 제한될 수 있다.

이는 현재 정의가 **세션 내 중복 증가 방지**라면 정상 동작이지만, 성공 지표에 사용할 **사용자별 순조회수**를 의미한다면 설계 보완이 필요한 상태다. 후속 검토에서 프로젝트별 일간 순열람으로 정의했으며 상세 정책은 `DECISIONS.md`의 ADR-012를 따른다.

### 미검증 범위

- 신규 회원가입과 이메일 인증
- 최초 로그인 사용자의 온보딩 화면 및 동의 저장
- 서로 다른 두 계정 사이의 작성자 권한 격리

기존 테스트 계정이 이미 온보딩을 완료했고, 추가 계정 생성과 이메일 발송은 명시적 승인 없이 진행하지 않았기 때문에 이번 범위에서 제외했다.

### 데이터 정리

검증용으로 생성한 프로젝트는 모두 삭제했다. 테스트를 위해 영구적으로 남긴 프로젝트 데이터는 없다.

### 후속 작업

1. 별도 테스트 계정 사용 승인을 받은 뒤 회원가입·이메일 인증·온보딩과 계정 간 RLS를 검증한다.

## 2026-07-07 일간 순조회수 적용 결과

- `project_views` 테이블과 새 RPC를 원격 Supabase에 적용했다.
- 실제 anon 호출에서 최초 호출은 집계되고 같은 익명 ID의 두 번째 호출은 집계되지 않았다.
- Selenium 상세 페이지 최초 진입은 `+1`, 하드 리로드 2회는 `+0`이었다.
- 로그인한 작성자가 본인 프로젝트를 열고 새로고침해도 조회수가 증가하지 않았다.
- anon·authenticated 역할의 `project_views` 직접 조회는 차단됐다.
- 기존 조회수와 검증 중 생성된 조회 기록을 초기화한 뒤 새 정책의 집계를 시작했다.

초기화 직후 열린 상세 페이지의 정상 첫 열람이 새 기준 조회수 `1`로 집계됐으며, 이를 새 정책 적용 이후의 유효한 첫 데이터로 유지했다.
